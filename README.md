# my-starter

A production-leaning TypeScript starter. Bun monorepo, Hono backend, TanStack Start web, Drizzle + Postgres (Supabase), Better Auth, oRPC end-to-end types, Resend email, varlock env, shadcn UI.

## Stack

| Layer | Tool |
|---|---|
| Runtime / package manager | [Bun](https://bun.sh) |
| Monorepo orchestration | [Turborepo](https://turborepo.com) |
| Web app | [TanStack Start](https://tanstack.com/start) + [TanStack Router](https://tanstack.com/router) + [TanStack Query](https://tanstack.com/query) |
| Server | [Hono](https://hono.dev) on Bun |
| RPC | [oRPC](https://orpc.unnoq.com) (contract-first, TanStack Query integration) |
| ORM / Database | [Drizzle](https://orm.drizzle.team) + [postgres-js](https://github.com/porsager/postgres) → [Supabase Postgres](https://supabase.com) |
| Auth | [Better Auth](https://www.better-auth.com) (email/password + verification + password reset) |
| Email | [Resend](https://resend.com) |
| Env management | [varlock](https://varlock.dev) |
| UI components | [shadcn/ui](https://ui.shadcn.com) (base-nova style, hugeicons) |
| Styling | [Tailwind v4](https://tailwindcss.com) |
| Linter / formatter | [Biome](https://biomejs.dev) via [Ultracite](https://www.ultracite.ai) |
| Git hooks | [Lefthook](https://lefthook.dev) |

## Prerequisites

- [Bun](https://bun.sh) ≥ 1.3.12
- A Postgres database. We recommend [Supabase](https://supabase.com) (the schema assumes you connect as the `postgres` role to bypass RLS — the direct connection or pooler with DB password).
- Optional: a [Resend](https://resend.com) account for real email sending. Without it, emails log to the server console in dev.

## Quick start

```bash
# 1. Clone + install
git clone <repo-url>
cd my-starter
bun install         # Runs `prepare` → installs hooks + generates env.d.ts

# 2. Configure env
cp .env.example .env
# Fill in DATABASE_URL and BETTER_AUTH_SECRET (see .env.example for format)

# 3. Push schema to your database
cd packages/db
bun --env-file=../../.env x drizzle-kit push
# Then in the Supabase SQL editor: paste & run `packages/db/src/setup.sql`

# 4. Start dev
cd ../..
bun dev             # Server on :3000, web on :3001
```

## Env management (varlock)

`.env.schema` is the single source of truth — typed, validated, and used to generate `env.d.ts` (gitignored; regenerated by `bun install` or `bun env:typegen`).

- **Server** (`apps/server`): `varlock run --path ../.. -- bun …` in scripts loads and validates env before spawning Bun. Code reads via `process.env.X` (typed by the global `ProcessEnv` declaration in `env.d.ts`).
- **Web** (`apps/web`): `varlockVitePlugin({ ssrInjectMode: "resolved-env" })` injects validated env at build/dev time. Code reads via `import { ENV } from "varlock/env"` — the plugin replaces `ENV.X` with the resolved value. Non-sensitive vars are exposed to the browser; sensitive ones (`@sensitive`) are server-only.
- **Bun**: `bunfig.toml` has `env = false` so Bun doesn't re-load `.env` after varlock has validated it.
- **Turbo**: `turbo.json` declares `globalEnv` so env vars reach task processes (Turbo 2+ is strict by default).

Adding a new var:
1. Add the line + decorators to `.env.schema` (`@type`, `@sensitive`, `@required`).
2. Add the value to `.env`.
3. `bun env:typegen` regenerates types.
4. Use it via `process.env.X` (server) or `ENV.X` from `varlock/env` (web).
5. Add it to `turbo.json` `globalEnv` so Turbo passes it through.

## Auth flow (Better Auth)

- **Email/password** enabled (`packages/auth/src/server.ts`). `requireEmailVerification: false` by default so dev UX isn't blocked; flip to `true` for production.
- **Email verification on signup** (`sendOnSignUp: true`). Click the link → auto-signs in.
- **Password reset** via `sendResetPassword` hook.
- **Email sender** (`packages/auth/src/email.ts`):
  - Real send when `RESEND_API_KEY` is set.
  - Falls back to `process.stdout.write` in dev when the key is empty — signup/reset flows still work locally without Resend credentials.
  - Idempotency keys derived from `${userId}/${sha256(url).slice(0,16)}` prevent duplicate sends for the same token (Resend best practice).
- **Session** stored as HttpOnly cookie. Cross-origin between web (`:3001`) and server (`:3000`) works via CORS `credentials: true` + `fetch(..., { credentials: "include" })` baked into the oRPC client.

## API (oRPC)

Contract-first. The `contract` defined in `packages/api/src/contracts/` is the source of truth for both server handlers and the browser client.

- **Procedures**: `publicProcedure` (no auth) vs `protectedProcedure` (requires `session.user` via middleware in `packages/api/src/middleware/procedures.ts`).
- **Context** (`packages/api/src/middleware/context.ts`): per-request object carrying `{ session, db, checkHealth }`. Built by `createContextFactory` and injected by the Hono oRPC middleware in `apps/server/src/app.ts`.
- **Router shape** (`packages/api/src/routers/index.ts`) composes feature routers:
  - `health` — service status probe (public)
  - `posts` — `listPublished` (public), `get` (public; 404 on unpublished unless owner), `listMine` + `create` + `update` + `delete` (protected, owner-only)

## Database security model

The starter is **server-only** — the browser never touches Postgres directly. Still, defense-in-depth is enabled:

1. **Row-Level Security** (`.enableRLS()` on every `pgTable`). With no `pgPolicy` defined, Postgres default-denies. Your backend bypasses this because `DATABASE_URL` connects as the `postgres` role.
2. **Revoked privileges** (`packages/db/src/setup.sql`, run once in Supabase SQL editor). Revokes all privileges on the `public` schema from `anon` and `authenticated` roles — closes the PostgREST attack surface that Supabase auto-exposes.
3. **Ownership checks** live in the service layer (`packages/api/src/services/posts.ts`). Updates/deletes filter `WHERE user_id = :session_user_id` AND assert ownership beforehand.

If you ever expose the Supabase anon key to the browser (Supabase client SDK, Realtime, Storage), switch from default-deny to explicit `pgPolicy` rules per table.

## Deploying

This starter is runtime-agnostic — Hono runs on Bun, Node, or Cloudflare Workers; TanStack Start builds to any static host + SSR runtime. A concrete deploy config is out of scope for the starter; recommended path is Cloudflare Workers (via Wrangler / Alchemy) with the same Supabase Zurich database.

For GitHub Actions deploy workflows, use `secrets.*` for sensitive values (`DATABASE_URL`, `BETTER_AUTH_SECRET`, `RESEND_API_KEY`) and plain `env:` for public config (`BETTER_AUTH_URL`, `BETTER_AUTH_TRUSTED_ORIGIN`). CI build (this repo's `ci.yml`) uses placeholder values inline — no real secrets are needed for `vite build`.

## Conventions

- **Conventional commits** (lowercase: `feat:`, `fix:`, `chore:`, `refactor:`).
- **Flat-by-kind** in `@repo/api`: features split across `contracts/`, `routers/`, `services/`. Service layer is opt-in per feature (health doesn't need one; posts does).
- **Ultracite / Biome** enforces style. `env.d.ts`, `routeTree.gen.ts`, and `packages/ui/src/components/**` are ignored (generated / upstream).
- **No AI attribution** in commit messages.
